# Known Issues

> Tracked items in the blst-java package (DCC fork of Decentral-America/blst
> / supranational/blst v0.3.15). Each item includes its risk level, reason it
> is not resolved immediately, and the recommended resolution path.

---

## KNOWN-1: All Java source files are SWIG 4.4.1 generated — do not modify by hand

**Risk:** INFO (process constraint — no security impact)

**Description:** Every `.java` file under
`src/main/java/supranational/blst/` is generated by SWIG 4.4.1 from the
upstream blst `.i` interface file. Hand-editing these files will be silently
overwritten the next time SWIG regenerates them.

The files are: `blstJNI.java`, `blst.java`, `P1.java`, `P1_Affine.java`,
`P2.java`, `P2_Affine.java`, `PT.java`, `Pairing.java`, `Scalar.java`,
`SecretKey.java`, `BLST_ERROR.java`.

**Why not changed:** Upstream blst uses SWIG as the authoritative JNI binding
generator. Keeping SWIG-generated code unmodified ensures zero divergence from
upstream and simplifies future upgrades.

**Resolution path:**
- To add Java-level behavior, wrap `blst.java` / `P1.java` etc. in new
  `io.decentralchain.blst.*` classes that delegate to the SWIG types.
- SpotBugs analysis of SWIG-generated files is suppressed via
  `config/spotbugs-exclude.xml` (false positives from generated code).
- Tracked in DCC roadmap for the next blst upstream upgrade sprint.

## KNOWN-2: Upstream blst v0.3.15 — latest upstream is v0.3.16 (unreleased JNI wrappers)

**Risk:** LOW (functionality gap — no security regression vs v0.3.15)

**Description:** The upstream supranational/blst repository has merged
changes beyond v0.3.15, including optimized assembly for ARM and minor
API additions. The Decentral-America/blst fork (our base) does not track
upstream HEAD continuously.

**Why not fixed now:** Upgrading blst requires regenerating SWIG bindings,
re-building native libraries for all 7 platform/arch combinations, and
re-running the full blst test suite. Deferred to a dedicated upgrade sprint.

**Resolution path:**
- Bump blst upstream to latest tag
- Regenerate SWIG bindings with SWIG 4.4.1 (or later)
- Rebuild all native `.so` / `.dylib` / `.dll` libraries
- Update version to `0.3.16.0` following DCC convention
- Tracked in DCC roadmap

## KNOWN-3: Native library loading uses a temporary directory

**Risk:** INFO (operational — no security issue)

**Description:** `blstJNI.java` extracts the bundled native library to a JVM
temp directory at startup (`Files.createTempDirectory("blst")`). On some
hardened systems with `noexec` on `/tmp`, this may fail.

**Why not fixed now:** This is the standard JNA/JNI pattern for bundling
multi-platform native libraries in a JAR. An `noexec /tmp` mount is unusual
and would also break many other JVM libraries.

**Resolution path:**
- If deployment requires `noexec /tmp`: set `java.io.tmpdir` to an
  exec-capable path, or pre-install the native library and use
  `System.loadLibrary("blst")` via `-Djava.library.path`.
